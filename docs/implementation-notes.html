Implementation Notes

项目：词源奇旅 / English Word Stories

此文件用于持续记录实现如何影响规格，或规格如何影响实现。
最近更新：2026-06-29

1. 设计决定

  • 已决定缓存键采用规范化小写单词。 输入大小写差异不影响缓存命中，展示时再转为首字母大写。
  • 已决定归档按单词分目录存储。 路径形如 archives/<word>/YYYYMMDD-HHMMSS--<word>.md，便于定位最新结果。
  • 已决定仅新生成结果写入归档。 读取缓存不会重复生成新归档文件，避免把“读取”误记成“生成”。
  • 已决定UI 采用 Claude 风格气质，而非像素级复刻。 规格未要求完全仿制，因此当前实现强调简洁、留白、柔和面板与阅读体验。
  • 已决定DeepAgent 从工程内 /skills 目录加载 skill。 这样 skill 与应用代码一起维护。
  • 已决定模型连接信息从 .env 读取。 当前支持 provider 专属变量（如 OPENAI_API_KEY、OPENAI_BASE_URL）以及通用兜底变量（LLM_API_KEY、LLM_BASE_URL）。
  • 已决定FilesystemBackend 显式设置 virtual_mode=True。 这样可消除 deepagents 的兼容性 warning，并避免依赖默认值变化带来的路径语义风险。
  • 已决定新增 provider 采用“通用前缀规范化”方案。 任何 provider:model 都会优先读取 {PROVIDER}_API_KEY / {PROVIDER}_BASE_URL，其中 provider 会被规范化为大写下划线，如 google-genai → GOOGLE_GENAI。

2. 对规格的理解 / 可能偏离点

  • 规格解释“优先查询存档的最新生成文件” 被实现为默认直接返回最新归档，并额外提供“强制重新生成”开关。
  • 规格解释“metadata、创建时间等元信息” 当前至少包含应用名、skill、word、display_word、cache_key、model、created_at、duration_ms、archive_filename。
  • 可能偏离当前 UI 不是多轮聊天界面。 它是单词查询工具页，因为规格重点在查询、模型、生成时间与结果展示。
  • 可能偏离词源内容质量依赖模型。 代码已接入 DeepAgent，但最终内容深度受所选模型与 API Key 可用性影响。
  • 规格解释“显示使用的模型” 现进一步落实为前端可视化模型配置区：用户不必手写完整的 provider:model，而是通过 provider + model name 组合生成。
  • 规格解释“页面核心主体应该是英文单词” 本轮调整后被落实为：模型/provider 配置折叠进“高级设置”，默认把视觉焦点让给英文单词输入框和结果内容区。

3. 折衷考量

  • 方案比较文件归档缓存 vs 数据库缓存。 当前选文件归档直读，因为需求本身就要求 Markdown 归档，最简单透明。
  • 方案比较FastAPI + 静态前端 vs 全栈前端框架。 当前选更轻的实现，优先完成 agent、缓存、归档主链路。
  • 方案比较纯 Markdown vs frontmatter + 正文。 当前选 frontmatter，既可读又易解析。
  • 方案比较固定模型 vs 自定义模型。 当前允许输入模型字符串，便于开发和试验。
  • 方案比较只读 provider 默认环境变量 vs 应用自行封装 .env。 当前选择在应用侧统一解析环境变量，原因是这样可以支持通用网关变量与 provider 专属变量共存。
  • 方案比较维护 provider 白名单 vs 通用规则。 当前从“半白名单校验”改为“通用前缀规则 + 特殊 provider 定制构造器”，这样新增 provider 时通常无需改后端代码，只需安装对应 LangChain 集成并配置环境变量。
  • 方案比较单一自由文本模型输入 vs 结构化 provider/model 输入。 当前选择“provider 下拉 + model name + custom provider”混合方案，以保留灵活性的同时降低配置错误率。
  • 方案比较始终展开模型配置 vs 折叠为高级设置。 当前选择默认折叠，以减弱 provider/model 配置在页面上的视觉权重，避免压过核心的单词输入与结果阅读。

4. 未决问题

  • 待确认缓存是否需要有效期？ 目前永久有效。
  • 待确认是否需要归档详情页 / 下载 Markdown？ 当前只显示归档路径。
  • 待确认是否需要流式输出？ 当前为一次性返回，更像工具页而非 Claude 实时生成体验。
  • 待确认模型输入是否应改为固定选项？ 当前是自由文本。
  • 待确认是否需要更严格 HTML 安全清洗？ 如果未来公网部署，建议补 sanitizer。

5. 实现变更记录

  • 2026-06-27 修正 Markdown 渲染路径。 之前实现会先对 Markdown 全量 HTML 转义，再交给 Markdown 渲染器，导致标题、列表、引用无法按预期显示。 现已改为直接渲染 Markdown。此改动提升了结果展示正确性，但也意味着若将来面向不可信输入，需补 HTML 清洗。
  • 2026-06-27 新增 .env 可配置 API Key 与 API 端点。 目前实现支持通用变量 LLM_API_KEY、LLM_BASE_URL，以及 provider 专属变量如 OPENAI_API_KEY、OPENAI_BASE_URL、ANTHROPIC_API_KEY、ANTHROPIC_BASE_URL。 这使应用可对接官方端点，也可对接兼容网关。
  • 2026-06-27 修正配置加载顺序。 由于默认模型 DEEPAGENT_DEFAULT_MODEL 需要在模块导入时读取， 因此将 load_dotenv() 前移到配置模块，避免出现“.env 已填写但默认模型未生效”的问题。
  • 2026-06-27 显式指定 FilesystemBackend(virtual_mode=True)。 用户运行时看到的 warning 来自 deepagents 对默认值即将变化的提醒，而不是 .env 本身出错。 当前已显式声明该参数，以稳定路径行为并消除 warning。
  • 2026-06-27 新增 .gitignore 并初始化 Git 仓库。 当前忽略虚拟环境、.env、构建产物、缓存目录及运行生成的归档文件，仅保留 archives/.gitkeep。 这样既能保护本地密钥与生成物，又能保持仓库干净可提交。
  • 2026-06-27 准备首个 Git 提交并推送到 GitHub。 这是工程交付层面的变更，不影响产品规格，但会影响版本管理与后续协作方式，因此一并记录。
  • 2026-06-29 将 provider 接入改造为通用可扩展实现。 旧实现只对少数 provider 做显式校验；新实现改为统一规范化 provider 前缀并自动读取对应环境变量。 这降低了后续接入新 provider 的代码改动频率，但也意味着某些 provider 的“缺少依赖/鉴权失败”将更多由 LangChain/provider SDK 在运行时抛出。
  • 2026-06-29 升级模型配置 UI。 前端从单一模型文本框升级为 provider 下拉 + model name 输入 + 自定义 provider 字段，并实时显示拼接后的模型标识及建议配置的环境变量名。 这降低了用户手写 provider:model 时的出错概率，但也引入了一套前端拼接规则，因此需要保证与后端规范化逻辑保持一致。
  • 2026-06-29 修复最近归档点击时的前端空引用风险，并收缩模型配置区。 当前为 provider/model 相关 DOM 操作增加了空值保护，避免某些前端状态下因元素缺失导致的 null.value 错误； 同时将模型配置折叠进“高级设置”，使英文单词输入框重新成为页面主视觉入口。
  • 2026-06-29 补全前端模型控件读取路径的空值保护。 在继续排查“最近归档点击后仍报 cannot read properties of null (reading 'value')”时， 发现 buildModelIdentifier() 仍直接读取了 modelNameInput.value。 当前已统一改为安全读取函数，避免同类问题在 provider/model 控件链路上再次出现。
  • 2026-06-29 为前端静态资源添加版本参数，规避浏览器缓存旧 JS。 在本地模拟测试已通过、但用户端仍持续出现旧错误时，判断高概率是浏览器仍在使用缓存的 /static/app.js。 当前已为 CSS/JS 资源追加版本参数，以确保页面刷新后拉取新版脚本。
  • 2026-06-29 为当前工作目录配置 Git safe.directory，并提交/推送本轮改动。 这是为解决本机 Git 的 dubious ownership 检查而采取的环境级修复，不影响产品功能，但会影响本地后续 Git 操作是否可用。
约定：后续任何会影响规格理解、交互行为、数据结构或安全边界的改动，都应继续追加到本文件。
